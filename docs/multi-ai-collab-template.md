# おまつさん専用・複数AI共同開発の運用テンプレ

## 目的
シエル / Claude Code / Hermes Code を同じ案件で安全に共同作業させるための、GitHubベースの運用テンプレ。

---

## 1. 基本方針

### 役割の考え方
- **シエル**: 司令塔、要件整理、タスク分解、レビュー観点整理、最終統合補助
- **Claude Code**: 主実装、構造整理、機能追加、バグ修正
- **Hermes Code**: 別案提案、レビュー、テスト観点、改善提案、比較材料作成

### 大前提
- 3者を「同じ会話部屋で完全自律協業」させるより、**GitHubを共通作業場にする方が現実的**
- 同じファイルを同時に触らせすぎない
- 最終判断者は **おまつさん or シエル** に固定する

---

## 2. repo設計テンプレ

### 案件repoの例
- `client-会社名-案件名`
- 例: `client-replus-sales-qa-chatbot`
- 例: `client-tenporescue-meta-lp`

### 推奨ディレクトリ構成
```text
project-repo/
├─ README.md
├─ docs/
│  ├─ overview.md
│  ├─ requirements.md
│  ├─ decisions.md
│  └─ prompts/
│     ├─ claude-code.md
│     ├─ hermes-code.md
│     └─ ciel-coordinator.md
├─ src/
├─ tests/
├─ examples/
├─ data/
│  └─ sample/
├─ .gitignore
└─ CHANGELOG.md
```

### 各ファイルの役割
- `README.md`: プロジェクト概要
- `docs/requirements.md`: 目的、要件、制約
- `docs/decisions.md`: 方針決定メモ
- `docs/prompts/`: AIごとの指示テンプレ保存
- `CHANGELOG.md`: 重要変更の要約

---

## 3. ブランチ命名ルール

### 基本形
- `main` : 正式版
- `feature/...` : 新機能
- `fix/...` : 修正
- `docs/...` : ドキュメント
- `test/...` : テスト関連

### AI別にわかるようにする例
- `claude/feature-initial-ui`
- `claude/fix-api-error`
- `hermes/review-auth-flow`
- `hermes/feature-alt-parser`
- `ciel/docs-requirements-v1`

### おすすめルール
**AI名 / 作業種類 / 内容**

例:
- `claude/feature-lead-form`
- `hermes/review-copy-logic`
- `ciel/docs-task-breakdown`

## 6. AIへの指示テンプレ

## A. Claude Code用テンプレ
```md
あなたは主実装担当です。

## 目的
[ここに目的]

## 今回やってほしいこと
- [作業1]
- [作業2]

## 完成条件
- [条件1]
- [条件2]

## 触ってよいファイル
- [パス]

## 触らないでほしいファイル
- [パス]

## 制約
- 既存構造を大きく壊さない
- 不要なリネームをしない
- 変更理由がわかるようにする

## 出力してほしいもの
- 変更内容の要約
- 注意点
- 追加で必要な作業
```
```

## B. Hermes Code用テンプレ
```md
あなたは比較・改善担当です。

## 目的
[ここに目的]

## 今回やってほしいこと
- Claude案の改善点の指摘
- 別実装案の提案
- 保守性/速度/安全性の観点からのレビュー

## 見てほしい対象
- [PRやファイル]

## 観点
- シンプルさ
- バグリスク
- 拡張性
- 実務運用しやすさ

## 出力してほしいもの
- 問題点
- 改善案
- 代替実装案
- どちらを採用すべきかの理由
```
```

## 7. 実務フロー

### 最もおすすめの流れ
1. **シエルに要件整理を依頼**
2. シエルがIssue化・タスク分解
3. Claude Codeに主実装を依頼
4. Hermes Codeにレビュー/別案を依頼
5. GitHubのPRで差分確認
6. シエルが比較整理
7. おまつさんが最終判断
8. mainへマージ

---

## 8. 役割分担の例

### 例1: LP改善案件
- シエル: 要件整理、訴求整理、改善観点整理
- Claude Code: LP修正実装
- Hermes Code: コピー改善案、UI/導線レビュー

### 例2: AIチャットボット案件
- シエル: 仕様整理、会話要件定義、FAQ整理
- Claude Code: 実装、API連携
- Hermes Code: 回答品質改善案、異常系レビュー、テスト観点出し

### 例3: SNS自動化ツール
- シエル: ワークフロー設計、プロンプト整理
- Claude Code: 自動化処理実装
- Hermes Code: 出力品質改善、保守性レビュー

---

## 9. やってはいけない運用

- 同じファイルを複数AIに同時に大きく触らせる
- 目的が曖昧なままAIに丸投げする
- 仕様書なしで実装から入る
- PRなしでmainに直接入れる
- AIごとに違う前提条件を渡す

---

## 10. 最初の最小セット

最初はこれだけあれば回し始められる。

- GitHubの案件repo 1つ
- README
- requirements.md
- Issue 3〜5本
- Claude用指示文
- Hermes用レビュー依頼文
- ブランチ運用ルール

---

## 11. おまつさん向けのおすすめ初期運用

### 小さく始めるなら
1. 1案件だけrepoを作る
2. シエルに要件整理をさせる
3. Claude Codeで1機能作る
4. Hermes Codeにレビューさせる
5. PRで比較する

### 最初に向いている題材
- LP改善の一部機能
- 簡単な問い合わせ整理ツール
- SNS投稿案の整形スクリプト
- FAQ整理用のミニツール

---

## 12. ひとことで運用原則

**会話を共有するより、仕様・タスク・変更履歴を共有する。**

これが複数AI共同作業を安定させるコツ。
